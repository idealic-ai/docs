# 21: Компания/Предложение

> [!DEFINITION] Документ Предложения
> Временный артефакт предложения, используемый для описания желаемого изменения, улучшения или расширения до начала его реализации.
>
> Sidenote:
> - Часть: :term[02: Компания/Процесс]{href="./02_process.md" canonical="02: Company/Process"}
> - Следующий этап: :term[22: Компания/Согласование]{href="./22_document_alignment.md" canonical="22: Company/Alignment"}
> - Ранее известен как: Документ о Намерении, Документ Эволюции

## 1. Основная философия

Документ Предложения, по сути, является **RFC (Request for Comments)**. Он пишется **один раз** для каждой инициативы, чтобы обозначить желание перемен. Он возникает потому, что мы не можем написать Спецификацию за один присест; мы должны разбить проблему на шаги. Одно Предложение примерно соответствует **Одному Акту** работы.

Он **сохраняется**. Он служит исторической записью изменения видения и становится **Источником Истины о Намерении**.

## 2. Необходимость

Зачем мы пишем Документ Предложения вместо того, чтобы сразу кодировать?

- **Преодоление «чистого листа»:** Мы не можем автоматизировать человеческое намерение. Предложение заставляет нас извлекать мысли из подсознания. **Диктовка** — предпочтительный метод для преодоления ступора и создания первоначального контента.
- **Источник задач:** Этот документ является отправной точкой, из которой создаются конкретные инженерные задачи. Это гарантирует, что работа разбивается на атомарные, управляемые части (например, однодневные задачи).
- **Разделение ответственности:** Он отделяет «Что мы хотим» от «Как мы это реализуем». Это позволяет нам обсуждать ценность идеи, не увязая в синтаксисе.
- **Безопасное исследование:** Он позволяет предлагать радикальные изменения, не нарушая источник истины. Если Предложение отклонено, теряется только текстовый файл.
- **Контекст для машины:** LLM требуют точного контекста. Этот документ предоставляет «инструкцию-промпт» для машины (и для команды).

## 3. Структура

Документ Предложения обычно содержит:

1.  **Контекст:** «Мы думали X, но узнали Y».
2.  **Проблема:** Что сломано, отсутствует или неэффективно?
3.  **Границы (Негативное пространство):** Четко укажите, **чего мы НЕ делаем**. Это предотвращает разрастание рамок проекта (например, «Мы исправляем уровень представления, а НЕ перестраиваем архитектуру бэкенда»).
4.  **Предлагаемое решение:** Высокоуровневый обзор изменения.
5.  **Варианты:** Альтернативные подходы (если применимо).
6.  **Цель:** Конкретные файлы или модули в Спецификации, которые будут затронуты.

## 4. Жизненный цикл

1.  **Черновик:** Создается автором. **Совет:** Надиктуйте свои мысли, чтобы быстро создать первый черновик.
2.  **PR:** Отправляется как Pull Request.
3.  **Обсуждение:** Обсуждается командой.
4.  **Доработка:** Обновляется на основе контекста из :term[22: Компания/Согласование]{href="./22_document_alignment.md" canonical="22: Company/Alignment"}.
5.  **Устаревание:** Сохраняется как история, но заменяется Спецификацией.

## 5. Характеристики

- **Сроки:** Создается примерно за 1 день.
- **Обновления:** Пишется один раз, чтобы зафиксировать изменение видения.
- **Природа:** Сохраняемый/Исторический.
- **Источник Истины:** **Для Намерения**.
- **Аналогия:** RFC / Один Акт.
