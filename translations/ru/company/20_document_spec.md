# 20: Компания/Спецификация

> [!DEFINITION] Спецификация
> Постоянный, развивающийся источник истины, который определяет видение продукта, архитектуру и детали реализации.
>
> Sidenote:

> - Часть: :term[02: Компания/Процесс]{canonical="Company/Process" href="./02_process.md"}
> - Предшествует: :term[22: Компания/Согласование]{canonical="Company/Alignment" href="./22_document_alignment.md"}

## 1. Основная философия

Спецификация — это **конечная цель**. Это результат большой и кропотливой работы — документ, который автор создает, улучшает и постоянно обновляет. Она не пишется за один день; это результат недель глубоких размышлений и множества итераций.

Она **не устаревает**. Это абсолютный **Источник Истины для Системы** (кода, презентации, архитектуры). В системе, где мы создаем сложных, автономных агентов, мы не можем полагаться на неформальные знания или временные переписки.

**Видение:** Мы верим, что при наличии правильной и полной Спецификации остальную работу (генерацию кода, дизайн презентаций) можно автоматизировать. Мы пишем этот документ, чтобы в конечном итоге «скормить» его машине.

## 2. Необходимость

Зачем мы поддерживаем этот документ?

- **Следование Истине:** Она строго следует стандартам, определенным в :term[00: Компания/Истина]{canonical="Company/Truth" href="./50_prompt_truth.md"}. Она должна быть самодостаточной и не допускать двусмысленности.
- **Корректность важнее скорости:** В отличие от временных документов (Предложение/Согласование), Спецификация должна быть абсолютно точной. Лучше оставить раздел неопределенным, чем определить его неверно.
- **Карта:** Она описывает, _куда мы движемся_. Она позволяет каждому члену команды видеть общую картину, не держа весь план в голове.
- **Управление сложностью:** Для больших систем с «широкой зоной охвата» невозможно удержать весь контекст в памяти. Спецификация позволяет системе развиваться, сохраняя единый источник истины.

## 3. Правила ведения

1.  **Никакого застоя:** Если меняется код, меняется и документ. В идеале, документ меняется _до_ кода.
2.  **Расширение через ревью:** Спецификация никогда не бывает «закончена», она всегда «актуальна». Во время ревью мы выявляем слепые зоны. Работа с ними расширяет охват документа, чтобы достичь ясности, необходимой для выполнения.
3.  **Источник Истины:** Если возникает конфликт между кодом и Спецификацией, документ считается верным, а код — ошибочным (либо документ требует обновления через процесс Предложения).

## 4. Соответствие по областям

«Система» — это не всегда код. Спецификация адаптируется к своей области:

| Область | Спецификация (Истина) | Артефакт (Результат) |
| :--- | :--- | :--- |
| **Разработка** | Архитектурный документ / Спецификация API | Код на TypeScript / Схема БД |
| **Презентация** | Структура повествования / Сценарий | Слайды / Figma |
| **Процесс** | Регламент | Действия команды |
| **Дизайн** | Сценарий пользователя / Каркас | UI-компоненты |

## 5. Характеристики

- **Сроки:** Недели на создание, постоянные обновления.
- **Этапы:** Множество итераций и доработок.
- **Роль:** «Большой документ», который служит опорой для проекта.
- **Природа:** Неустаревающий.
- **Источник Истины:** **Для Системы**.
